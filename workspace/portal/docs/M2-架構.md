## 3. BFF 端點契約（報告 §8 + 標註偏離）

```
GET /api/services      → { groups:[{group, items:[{name, url, url_hl?, reach, kuma_ok:bool|null, note?}]}], generated_at }
GET /api/security      → { autoban_today, autoban_trend_24h:[…], cowrie:{count, top_src:[{ip,geo,n}]} | {offline:true, hint},
                           tripwire:{last_hit, days_clean}, generated_at }
GET /api/power         → { pending:true, hint } 直到 nut_exporter 就緒；之後
                         { on_battery, charge, load, runtime_s, events_7d, generated_at }
GET /api/game          → { server_up, instance_state, players_online, player_names:[…]|null, generated_at }
GET /api/life          → { calendar_today:[{time,title}], debts_open:{count, total}, generated_at }
GET /api/host/{name}   → { metrics_6h:{cpu:[…],mem:[…],disk:[…],net:[…]}, services:[…],
                           related_alerts:[…], log_tail:[{ts,line}]|null, grafana_url, generated_at }
```

標註偏離報告 §8（理由攤開，隨 §7 一併裁決）：
- `/api/services` 頂層由陣列改物件——容納 generated_at，與 M1 契約慣例（Overview/Alerts/Brief 皆帶）對齊。
- `cowrie.count` 沿用報告欄名（不改名）；`top_src` 展開為 {ip,geo,n}、`autoban_trend_24h` 為增量補充。
- `/api/power` 的 `{pending:true,hint}` 形態為報告未定義的補充（§ 7 決策 2）。
- `/api/game`、`/api/life` 為報告 §8 未列的補充端點——§2「前端只跟 BFF 說話」+ §3 模塊表
  必然要求其存在（life 資料源見報告 §4「借貸/行程」列；game 資料源僅見 §3 模塊表，§4 未列）。

行為規範：
- 快取照報告 §4/§8 全部 **3–10s**（providers.py 現為單一 CACHE_TTL=5s，擴充為逐端點 TTL 但不超出 10s；
  前端輪詢週期 15–60s 已天然限流，長快取無收益）。
- **失敗 negative cache 30s**：上游持續失聯時不逐 tick 重打吃滿 timeout（SSE 迴圈 15s 一輪，
  夜間 CT270 關機屬常態，不能讓它拖慢每一輪）。
- `/api/services` **例外不 502**：清單本身是 BFF 內靜態表；Kuma 失聯時仍回 200、
  全部 kuma_ok=null（灰燈）+ note「Kuma 失聯」——服務目錄是退役 Homepage 後的日常唯一入口，
  不允許因探活源掛掉而整頁死。
- 上游回 401/403 → note 明示「憑證失效」，與 timeout/refused 區分（使用者一眼知道要輪替 key 而非查網路）。
- 其餘端點上游全掛 502 `{error,hint}`；部分掛 → 對應欄位 null/offline + note，端點仍 200。
- SSE 維持只推 overview/alerts（報告 §4 其餘源 30–60s/進頁拉，輪詢即可）。

## 10. 已知限制（設計時即接受，寫明免日後誤判）
