### 監控 — node-exporter 宿主數據污染修正

- 通則：監控 LXC「整體」資源，node-exporter 要原生跑（在容器主 cgroup），不要在容器內用 Docker。
  備份：docker-compose.yml.before-nodeexporter-fix-*、prometheus.yml.before-native-nodeexporter-*。

### IPv6 — relay 嘗試與結論（已全部還原）
- 試 odhcpd relay/ND-proxy（共用 wan6 on-link /64），下游能拿 GUA 但「出網逾時」。
- 根因：PPPoE 點對點上游讓 NDP relay 的 per-client 回程代理建不起來（PACKET_ADD_MEMBERSHIP: Invalid argument），
  router 對下游 GUA 查路由 = unreachable -> 回程被丟。
- 結論：HiNet 單一 /64 + PPPoE，relay 是死路，別再試。備份：{dhcp,network,firewall}.before-v6relay_20260630_125526。

### IPv6 — 最終方案 A：servers 段 DHCPv6 server 模式給真 GUA（已上線）
- 卡關真兇：幽靈介面 wan_6 無限 flap。根因鏈：network.globals.dhcp_default_duid 有設 ->
  客製 dhcpv6.sh 呼叫 hexdump_2hex（此函式全系統未定義）-> odhcp6c 參數錯 ->
  pppd AUTOIPV6=1 自動生成的 wan_6 setup 失敗 -> 每幾秒 flap -> 事件風暴 -> odhcpd 反覆重列舉、RA 計時器永遠重置 -> servers 收不到 RA。
- 修法：`uci set network.wan.ipv6='1'`（關 AUTOIPV6，不再自動生 wan_6；手動 wan6=proto dhcpv6 device @wan 照常）。
  reload 後 wan_6 幽靈消失、RA 立刻通（此 reload 會讓 wan6 重協商、GUA/PD 換新，PPPoE 本就會漂移，正常；tailscaled 自動發布新端點）。
- 生效設定：network.servers.ip6assign='64'；dhcp.servers ra=server dhcpv6=server ndp=disabled dhcpv4=disabled ra_mtu=1492。
  實測 CT260 拿到 GUA、curl -6 出網 30ms、公網回傳自身 GUA（無 NAT）。安全：wan input/forward DROP 不變，入站到 GUA 被擋。
  範圍：只有 servers(.20) 有 v6，其餘段維持無 v6（要加照同模式套）。
- dhcp_default_duid 已 uci delete 清除（本來就被 bug 廢掉沒生效）。
- 根因註：hexdump_2hex 缺失是 OpenWrt 25.12 上游已知 bug（本機 25.12.4 r32933 x86/64）：
  netifd /lib/netifd/proto/{dhcp,dhcpv6}.sh 引用但 build 未含其定義（應為 hexdump_2hex(){ hexdump -ve '1/1 "x"'; }，放 /lib/functions.sh）。
  只在設了 clientid 或 dhcp_default_duid 時才觸發。若未來要用固定 DUID（讓 HiNet 給穩定前綴），把 hexdump_2hex 補進 /lib/functions.sh 再設 duid。
  備份：network.before-wanipv6fix_20260630_140434、{dhcp,network}.before-v6serverA_20260630_133856。

### WiFi 手機 YouTube 首頁載不出 — iot QUIC(udp/443) 過小米 AP 壞 → 擋 QUIC 逼回 TCP（2026-07-04）
- 症狀：兩支 WiFi 手機（Abyss-de-S24-Ultra .50.89、vivo-X100 .50.108）YouTube 首頁載不出；
  有線 AppleTV/PC、手機關 WiFi 走行動網路皆正常；AGH 訪問記錄解析成功。
- 診斷（先查現況再改，逐一排除）：
  ① IPv6 假設（手機優先走壞掉的 v6）否決：dhcp.iot 無 ra/dhcpv6/ndp、br-iot 僅 link-local、iot v6 鄰居兩次皆空 → 手機根本無 v6。
  ② TCP MTU 黑洞否決：wan zone mtu_fix=1，nft 確認 pppoe-wan 進出雙向皆 MSS clamp（1492）→ 基於 TCP 的網頁 MTU 已保護。
  ③ 雙重 NAT 否決：兩手機都拿 OpenWrt 的 .50.x 租約 → 小米 AP 純橋接，br-iot MTU 1500 與有線一致。
- 根因：YouTube/Chrome 優先走 QUIC(HTTP/3, udp/443)；此 UDP 路經小米 AP 的 WiFi 會壞（大型 UDP 被丟/mangle，AP 側無可設定項）。
  DNS(udp53 小封包)與 TCP(有 MSS clamp)都正常，唯 QUIC 壞且瀏覽器不回退 → 首頁卡。有線/行動網路不經此 AP 故正常。
- 修法（OpenWrt 唯一改動，可逆）：firewall config rule
  Block-IoT-QUIC-udp443-To-WAN-YouTubeFix（src=iot dest=wan proto=udp dest_port=443 target=REJECT）。
  落點：forward_iot 內排在「accept_to_wan」之前（REJECT 先命中）；REJECT=ICMP port-unreach 使手機即刻回退 TCP。
  fw4 check PASS、reload OK。事後計數 184 pkts/258KB 被擋 → 坐實手機一直在打 QUIC。使用者實測 YouTube 恢復。
- 與 VLAN20 IPv6（2026-06-30 方案A）無關：時間差 4 天、不同段（iot vs servers）、手機無 v6、壞的是 IPv4 QUIC。
- 為何今天才發：OpenWrt 端無法定觸發點（開機 6 天未重啟，規則計數只證「進行中」非「何時起」）。
  最可能＝小米 AP 韌體/狀態變化開了硬體加速/QoS 弄壞大型 UDP，或 YouTube/Chrome 端 QUIC 更積極；此體質潛伏、今天才踩到。根治需動 AP，本規則為穩定迴避方案。
- 備份/回滾：OpenWrt /root/firewall.before-quic-iot-20260704-140231。
  回滾＝cp 回 /etc/config/firewall && fw4 reload，或刪該 named rule。
- 文件：拓樸V6 §4.1（iot→wan 註記）、§4.4（QUIC 擋條，SoT）同步。

### 待辦⑮追加 OCR 引擎換 RapidOCR sidecar（2026-07-05）
背景：使用者實測 tesseract chi_tra 對真實中文掃描件辨識很差（UI 截圖辨成「號 re S Pa eravie we」），
  指示換引擎（參考 pre-consume script 換引擎方案，評估後採 sidecar 變體）。
架構定案：ocr 容器（本地 build paperless-ocr:rapidocr3.9.1）＝RapidOCR 3.9.1（onnxruntime）＋
  FastAPI（POST /ocr 收 PDF 回雙層 PDF）＋consume watcher 執行緒（圖→雙層 PDF）。
  引擎不進 paperless 容器（每次 recreate 乾淨、pip 全家桶不可行）；models/fonts 隨 build context
  烘入映像 → 執行期與 rebuild 零外聯（rebuild 僅 pypi，永久白名單涵蓋）。
模型選型（本地 venv 實測三組合後定案）：PP-OCRv5 det=mobile＋rec=server。
  rec mobile 難字件漏字（陳大文→陳、甲方→甲、貳→贰、「自…8月1日…止」漏月日）；
  rec server 全對（另次 mobile 反而對統編 22555003 一位——server+300dpi 渲染管線下兩件全對）。
  CT270（2C）每頁約 6-10 秒、sidecar 常駐 ~708MiB（mem_limit 1536m）。
★ 踩坑1：BuildKit 拉 base image 不走 dockerd systemd proxy（dial 直連 refused）——先 docker pull
  python:3.12-slim（daemon 傳統路徑走代理）再 build --pull=false。
★ 踩坑2：squid dstdomain 子域重複＝parse FATAL（非只 WARNING）——暫窗行 .pythonhosted.org 與
  既有永久白名單 files.pythonhosted.org 相撞，squid 拒 reconfigure。教訓：加窗前先 grep 既有清單。
  （原永久白名單本就含 pypi.org files.pythonhosted.org，暫窗實際只需 .docker.io/cloudfront/.modelscope.cn。）
★ 踩坑3：pre-consume 對圖片不可換型——paperless 在 pre-consume 前已依原始 mime 選 parser，
  把 working file 從 JPG 換成 PDF 內容 → PIL「cannot identify image file」ConsumerError。
  解法：script 只轉發 PDF（頭 5 bytes 檢查）；圖片改由 ocr 容器 watcher 在 consume 資料夾先轉
  雙層 PDF（原圖刪、寫 X.pdf；X.pdf.tmp 過渡），webserver 以 PAPERLESS_CONSUMER_IGNORE_PATTERNS
  忽略圖片與 *.tmp。副作用：手機 app「分享原始照片」（API 直傳圖片）仍走 tesseract 後備；
  app 內建掃描（產 PDF）不受影響。
★ 踩坑4（兩層）：文字層寫入/萃取的字型陷阱——
  a. pymupdf 內建 CJK 字型（china-t）無可攜 ToUnicode：pymupdf 自己讀得回、pdfminer/paperless
     萃取成 CID 亂碼。→ 改 insert_text(fontfile=NotoSansTC) 嵌真字型＋doc.subset_fonts()。
  b. subset_fonts 對 OTF（CFF）炸「Reserved charstring byte」→ 整包 16MB 全嵌（PDF 13.9MB）。
     → 換 TTF（google/fonts NotoSansTC[wght].ttf），subset 後每份約 +40KB。
  c. 即便如此,ghostscript 產 PDF/A archive 時重寫 subset 字型 ToUnicode → 萃取出整齊 -0x100
     碼位錯字（送 U+9001→輁 U+8F01、錄 U+9304→鈄 U+9204）。→ PAPERLESS_OCR_SKIP_ARCHIVE_FILE=
     with_text（有文字層不產 archive，內文直取原件；tesseract 後備件無文字層仍照常產 archive）。
  殘留小疵（接受）：萃取端可能把 '-' 反查成 U+2011（NotoSansTC cmap 多碼位共 glyph），
  whoosh 斷詞後 AB／12345678 等詞元照樣可搜。
