## 1. CT260 Codex Ops

CT260 是乾淨的 Codex 操作節點，不是全權管理節點（基本資訊見拓樸 §10）。

```text
1. codex 使用者沒有 sudo，這是刻意設計。
2. 不要把 PVE root / OpenWrt root 私鑰複製到 CT260。
3. 若要讓 CT260 管理 homelab，應建 CT260 專用 SSH key + 最小 sudo allowlist，不要沿用舊 key 或全 sudo。
4. 用完臨時權限後要有撤銷 runbook。
5. DeepSeek key 可長期放 ~/.config/homelab/deepseek.env（codex 擁有, chmod 600；notifier 需要）；
   OpenAI / GLM key 仍維持「非必要不長期存放於 CT260」。
6. CT260 兼任 Ansible controller（待辦48 裁決，不建 VLAN90 host）：CT 一律走 pct_remote
   （ssh pve24 + sudo pct），勿為 Ansible 在任何 CT 內開 SSH。
7. OpenWrt sysupgrade 不保留自裝套件：升版後 python3 需重裝，否則 Ansible 對 openwrt
   只剩 raw 模組（piped 傳輸與 dropbear 無關，不受影響）。
```

## 3. OpenWrt DNAT / Hairpin

（DNAT 表見拓樸 §4.2、hairpin 見 §4.3）

```text
1. wan6.device 必須指向 @wan，不要錯綁 LAN interface。
2. LAN hairpin 是額外規則，不是 WAN DNAT 自然生效。
3. Minecraft hairpin 綁定公網 IP 125.228.123.224；PPPoE 公網 IP 改變時要同步更新，或改 split DNS/SRV。
4. 改 DNAT 前先 `nft list chain inet fw4 dstnat_wan`。
5. 改完 fw4 reload 後務必從 PC 與外部路徑實測。
6. OpenWrt 上的 samba4-server 是「刻意」裝的，只為 nmbd 當 iot 段 NBNS 代答者（門口相機掃 NAS 用，見拓樸 §4.4）。
   別把它當檔案伺服器、別擴大 uci samba4 interface（現為 'loopback iot'）；smbd 監聽 .50.1:445/139 是被 DNAT/input-REJECT 遮蔽的無害副產物，勿為此另開 iot→router 445/139。
   相機的 iot→storage DNAT（.50.1:445/139 → .30.3）在 dstnat_iot，非 dstnat_wan。
```

## 4. portscan-autoban / Cowrie quarantine

```text
1. 每日清空的是未達門檻 counter，不影響已封鎖 IP。
2. 不要把 44444 / 49169 / 43915 納入封鎖計算，否則正常服務會被誤傷。
3. Cowrie 小黑屋用獨立 nft table inet cowriequarantine，不要混入 portscanban set。
4. Cowrie 小黑屋 timeout 10 天；新登入嘗試會刷新該 IP timeout。
5. 小黑屋 IP 仍可連 WAN tcp/22,23（進 Cowrie），其餘 WAN 入口會被 drop。
6. 小黑屋流量速率圖只反映「已進小黑屋來源」的 drop/放行速率，不是全站 DDoS 偵測。
7. GeoIP 只是輔助，不應作為唯一判斷依據。
8. 可視化在 CT201 Prometheus/Grafana（見《監控告警.txt》）。
```

## 5. Honeypot（VM300）

```text
1. Honeypot 不應主動連內網服務。
2. Honeypot 不應使用可解析內網服務的 DNS，除非做了嚴格限制。
3. Cowrie 原始 JSON / TTY log 應留在 VM300 作證據源；CT201 只保 Prometheus 指標，不取代原始 log。
4. cowrie-exporter 只讀 cowrie.json；9116 只允許 CT201 scrape。
5. cowrie-geoip-exporter 在 CT201 跑（對 Cowrie Top IP 查 ip-api.com/batch）；VM300 不主動對外查 GeoIP。
6. VM300 PVE net0 必須保留 tag=666, firewall=0。
7. 要查完整攻擊 session/命令全文，未來再加 Loki/Promtail 或等價日誌管線。
8. PVE WebUI 記憶體用量跟 Grafana（vm300-honeypot-node）看起來差很大是正常現象，非監控故障：
   - VM300 是全宿主唯一一台真正的 KVM VM（其餘 CT 皆 LXC，不吃 balloon），宿主記憶體吃緊時
     automatic ballooning 只會壓它一個，一路壓到 `balloon:` 下限（現況 512MB）。
   - PVE WebUI 的 % = guest 已用量 ÷ `memory:` 設定上限（4096MB，分母是「潛在上限」）→ 數字看起來很低。
   - Grafana/node_exporter 的 % = (MemTotal−MemAvailable) ÷ MemTotal，而 MemTotal 本身已被 balloon
     同步壓縮到跟 balloon actual 差不多大（現況 ~338MB，分母是「目前實際拿到的量」）→ 數字看起來很高。
   - 兩邊算的分母不同，不是同一件事對不上；判斷 VM300 guest 內部是否真的緊繃要看 Grafana/node_exporter
     這條（MemAvailable），PVE WebUI 的百分比只能看「相對於上限」的粗略趨勢。
   - 2026-07-04 首次診斷時：宿主 24bay-pve 可用記憶體 17%、VM300 guest 內部可用僅 ~5%，
     两条 HostHighMemory 同時 firing；使用者選擇暫不處理（balloon 下限/宿主壓力來源皆未動），純觀察。
```

## 6. CT201 觀測面（VLAN80）

```text
1. CT201 是觀測面，不是管理面。允許「看」，禁止操作面（PVE 8006 / LuCI / AP/Switch Web / NAS admin /
   SMB·NFS write / Docker socket / DB write / SSH 到任意內網）。完整 observation policy 見《網段指導原則.txt》VLAN80。
2. 開觀測面跨段放行要「逐項點名」，不要一次性開整個管理面（blast radius 過大）。
3. 監控 LXC「整體」資源，node-exporter 必須「原生」跑（在容器主 cgroup），不要在容器內用 Docker
   —— 否則會繞過 lxcfs、把宿主機數據當成該 CT。（沿革見缺改記錄 2026-06-29）
4. 新增 scrape 前先確認 CT201 能直接 curl 到目標 /metrics，避免長期 down target。
5. 硬體溫度 / SMART 以 PVE host 層 node_exporter 為準；LXC 內 node_exporter 看該 CT 的 fs/process/memory 趨勢。
```

## 13. CT203 DMZ 反代 / tailnet 動態閘門

CT203 是 tailnet→內網的唯一把關點：OpenWrt 對 10.60.60.10 是「靜態 proto-all 常開、無後手」（fw4 forward_dmz 只匹配 saddr 即 accept），把關全在 CT203 nft `table inet ctdmz`。見拓樸 §4.8。

```text
1. ★ 改 ctdmz.nft 後必驗「閘門真的載回來了」，別只看 systemctl is-active：
   `systemctl restart ctdmz-nft.service` 後 → `nft list table inet ctdmz` 的 forward 鏈仍有 tailscale0 的
   reject+drop 兩條兜底，且 CT201 `ctdmz_gate_up==1`（每 30s 刷新，見監控告警.txt §3）。
   整表載入失敗＝fail-open（未驗證 tailnet 直穿內網），且是「靜默」的——CtdmzGateDown 告警就是為此而設。
2. restart ctdmz-nft.service 會清空 ts_authed_v4 通行證集（fail-closed，安全預設）→ 之後所有 tailnet→內網
   （Jellyfin/Navidrome 豁免除外）須重過 Authelia 才恢復。套用時機避開他人正在用；套完自己重驗一次內網服務可達。
3. 媒體免驗證豁免（destination-based、靠服務自身帳密，繞過 Authelia）目前有 Jellyfin 30.3:8096、Navidrome 20.70:4533。
   新增/放寬任何豁免＝擴大免 Authelia 面，須逐條使用者批准 + 更新拓樸 §4.8 +（可選）擴 gate 監控——
   現 `ctdmz_jellyfin_exemption_pinned` 只盯 8096，新豁免不在守護範圍。
4. CT203 主機自身 outbound 不受 ctdmz 管制（無 output 鏈）＋ OpenWrt proto-all → CT203 一旦被打穿即可 proto-all
   橫移 8 內段（安全模型＝完全信任 CT203）。故其公網攻擊面（Caddy/cloudflared/Authelia 容器）務必最小化；
   least-privilege 收斂選項（把 §4.4 那 8 條 proto-all 收成實際服務埠）見拓樸 §4.8 尾註 W2，未做。
```
