## 1. 架構

```text
1. 各機器/CT/VM/OpenWrt/exporter 持續暴露 metrics。
2. CT201 Prometheus(9090) scrape 所有目標。
3. Prometheus 載入 rule files(/etc/prometheus/rules/*.yml) 判斷異常。
4. Alertmanager(9093) 接 alerts，依嚴重度/類型發送。
5. 通知三軌（2026-07-07 分流，路由詳 §5g）：
   - Gotify(8090) 內網全收（observation plane 零對外）。
   - CT260 Telegram notifier 外送 warning/critical＝敘事頻道（告警文字+AI 分析+日報；egress 集中在 CT260）。
   - CT201 ntfy(8091)＝動作頻道：只收 critical FIRING（處置按鈕面；critical=TG+ntfy 唯一雙推例外）。
6. Grafana(3002) 可視化；Uptime Kuma(3001) 作外層服務探活（含 CT201/Prometheus/Gotify 自身，避免 SPOF 盲點）。
```

Telegram 走 CT260 而非 CT201 的原因：CT201 維持零對外；CT260 本來就是 AI/ops 節點、已有 outbound。
CT260 不能直連 VLAN80，故經 `ssh pve24 -> pct exec 201 -- curl localhost:9093` poll Alertmanager。
管線自身的第二層防護（待辦#4，2026-07-05 上線）：站外看門狗 watchdog-ct201（§5c）
+ notifier fetch 自檢（連續 3 次讀不到 Alertmanager 直發 TG，見 §5c 末）。

## 3. 告警規則現況（CT201 /opt/monitoring/rules/，共 34 條）

```text
alerts.yml   （核心）Watchdog（vector(1) 心跳）、PrometheusConfigReloadFailed、
             PrometheusRuleEvaluationFailing、TargetDown（up==0 for 3m）

hardware.yml （全 node_exporter，severity=warning）
             HostHighCPU(>60%/5m)、HostHighMemory(可用<30%/5m)、HostHighIOWait(>25%/5m)、
             HostDiskLowSpace(<10% free/15m，已排除 tmpfs/overlay/squashfs)、HostRebooted(boot time 變)、
             HostDiskWillFillSoon(predict_linear 24h 滿盤 且 <30% free/30m)、FilesystemReadWriteSpike(讀寫>60MB/s/15m)

services.yml （service_up + staleness）
             service_up==0：PortscanAutobanServiceDown(crit)、CowrieQuarantineServiceDown(crit)、
                            OpenWrtFirewallDown(crit, openwrt_service_up{service=firewall})、
                            OpenWrtServiceDown(warn 其餘)、CowrieExporterUnhealthy(warn)
             staleness：PortscanMetricsStale、CowrieQuarantineMetricsStale、
                        OpenWrtSystemMetricsStale、CowrieLogStale

security.yml （防火牆/蜜罐）
             PortscanBanListEmpty(banned_ips==0，絕對值避開每日 reset 誤報)、
             PortscanBanSpike(delta>80/10m)、CowrieQuarantineSpike(delta>30/10m)、
             SSH22TripwireHit(increase(openwrt_ssh22_tripwire_packets_total[5m])>0，critical，
               ai_analyze=true 首發即 Tier2——內網 22 死埠絆線，見拓樸 §4.5.1；
               告警用聚合 counter 非 per-src series（新 series 首現 increase() 不觸發的坑）；
               無白名單，內網掃描稽核前先 amtool silence)、
             SSH22TripwireInactive(openwrt_ssh22_tripwire_active!=1 for 10m，warning——絆線靜默失效守護)
             （蜜罐加固③ 2026-07-05 新指標隨 9105 同 endpoint：openwrt_honeypot_ratelimit_active gauge +
               openwrt_honeypot_ratelimit_dropped_packets_total{scope=per_src|aggregate} counter，

             CtdmzJellyfinExemptionDrift(ctdmz_jellyfin_exemption_pinned==0 for 5m，warning，ai_analyze=true——唯一免驗證豁免綁定異動)
```

## 5b. CT260 extension-host 空轉偵測 cpu-orphan-watch（本機獨立 cron，非 Prometheus）

程式：CT260 `~/.local/bin/cpu-orphan-watch.py`（codex 擁有），cron `*/30` + flock（`.cow.lock`）。
用途：抓 VS Code 遠端 `type=extensionHost` 因 dangling pipe(readable-on-EOF) 空轉的高 CPU（見《缺改記錄》2026-07-01）。
判定：取樣 4s，extensionHost 持續 CPU ≥ 門檻即告警；門檻預設 40%（env `COW_CPU_THRESHOLD`；另 `COW_STALE_MINUTES`=6h、`COW_COOLDOWN_SECS`=2h）。
輸出：寫 log 並發 Telegram（沿用同一 notify-telegram.env）；訊息附長存活 claude/codex 候選孤兒 PID + tmux 進程數。
cooldown 2h 不洗版、恢復正常補發一則。狀態/日誌：`~/.local/state/homelab-notify/{cpu-orphan-watch.log,cpu-orphan-watch.state.json}`。
處置指引：kill 告警中的殘留 session PID，或 VS Code → Developer: Restart Extension Host。

## 5c. CT201 站外看門狗 watchdog-ct201（監控的監控，本機獨立 cron，不經 Alertmanager；待辦#4，2026-07-05）

程式：CT260 `~/.local/bin/watchdog-ct201.sh`（codex 擁有），cron `*/2` + flock（`.wd201.lock`）。
用途：CT201 或其上告警要件死亡時仍能收到通知（Prometheus/Alertmanager 全滅時的最後一層）。
檢查四點（單次 ssh pve24 -> pct；CT260 不能直連 VLAN80，未動防火牆）：pct status 201 running、
  Prometheus :9090/-/healthy、Uptime Kuma :3001（實際回 302，curl -sf 成功即判活）、Gotify :8090/health。
防抖：連續 >=2 輪（約 4 分鐘）失敗才直發 TG（沿用 notify-telegram.env）；恢復補發一則；
  ALERTED 旗標保證告警一次不洗版。ssh pve24 不通 -> 四項全失敗（訊息標明 ssh-pve24不通）。
狀態檔 `/var/tmp/wd-ct201.state` 兼心跳（LAST_RUN/LAST_STATUS/FAIL_COUNT/ALERTED/FAIL_DETAIL）；
  life-ops daily_brief 讀之輸出【看門狗】行（>10 分鐘無心跳判停擺）。log：`~/.local/state/homelab-notify/watchdog-ct201.log`。
測試鉤子：`WD_SSH_HOST=<不存在主機>` 模擬全鏈失敗（2026-07-05 已實測 🚨告警+✅恢復雙向實收）。
盲區：CT260/pve24 自身死亡（CT260 與 CT201 同宿主，本鏈無法自救）——由 UPS 關機序 + 復電自啟涵蓋。
另一層自檢（待辦#4d）：homelab-notify.py poll Alertmanager 連續失敗 >=3 次 -> 直發 TG「告警管線異常」
  （只在跨門檻那次發一則），恢復發「告警管線恢復」並歸零；計數檔 `~/.local/state/homelab-notify/fetch_fail_count`。

DeepSeek 分層（成本感知）：

```text
Tier1（新 FIRING）：fast 模型給「一句話分流」（什麼節點什麼狀況、故障 or 被攻擊），附告警末（⚡）。
Tier2（連續 firing >= ESCALATE_AFTER_MIN=5m，只發一次）：deep 模型，
   呼叫前先撈 Prometheus「即時快照 + 近 30m 趨勢」餵入，輸出「數據判讀/定位/根因/影響/處置/可信度」（🔬）。
   告警中途 resolved 再 fire -> startsAt 變 -> 計時與 escalated 重置（必須連續燒滿才升級）。
ai_analyze:"true" label -> 首次 fire 就直接 Tier2。
同週期 >= AI_BATCH_THRESHOLD(3) 個新 FIRING -> 額外送一則關聯分析。
宿主告警（HOST_GUESTS，目前 pve24-server.home.arpa:9100）：deep/batch 額外附「底下各 CT/VM 即時占用」
   （CT100/102/201/260/VM300 node_exporter 快照；CT250 無 exporter 不列），定位哪台 guest 造成宿主壓力。
```

## 8. 設計/規劃參考（告警分層藍圖）

現況已覆蓋第一~四階段（見 §3）。以下保留為「告警分類藍圖」與後續方向。

```text
第五階段 長時間/並發複合判讀（用 recording rules 整理關鍵狀態後再建複合告警；這層才給 ai_analyze 或靠 batch）：
  - 高 CPU + 高 iowait + 高磁碟讀取 -> 磁碟壓力/備份/壞盤重試/異常程序。
  - 記憶體飆升 + swap 增加 -> CT/VM 服務記憶體洩漏。
  - 某 VM disk r/w 異常 + Cowrie/防火牆事件同步升高 -> 攻擊流量/log 爆量/exporter 重讀。
  - OpenWrt CPU 高 + 封禁數暴增 -> 掃描爆發/規則壓力。
  - 多個 exporter 同時 down -> 先疑網路/VLAN/OpenWrt/CT201/scrape 路徑，而非單點服務。
  - 單一內部設備長時間 unreachable -> 先查該設備/放行規則/VLAN tag/exporter 綁定 IP。
第六階段 Grafana 告警總覽面板：firing/pending（按 severity/instance/job）、硬件異常、服務狀態、防火牆安全總覽。
告警嚴重度分級：info=狀態變化/重啟/短暫異常；warning=持續異常但未必中斷；critical=監控核心/防火牆核心/主要服務不可用。
靜默策略：維護時段 silence；已知關機設備（如 CT250）避免常駐噪音。
```
